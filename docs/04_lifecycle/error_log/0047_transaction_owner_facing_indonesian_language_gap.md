# 0047 Transaction Owner-Facing Indonesian Language Gap

Status: Proposed

Reported by owner on 2026-06-27 while reviewing report readability work in
`0046`.

## Scope

Normalize transaction-facing UI language into clear Indonesian for text that is
generated by the system and shown to users.

This scope includes transaction and note-facing surfaces such as:

- transaction/note list and detail pages;
- create/edit transaction workspace labels generated by the system;
- payment, refund, revision, correction, and operational-state labels;
- report labels that describe transaction state;
- PDF/Excel headers where labels are system-generated;
- validation, empty-state, warning, action, and status copy generated by the
  application.

## Scope Out

Do not rewrite user-provided input values:

- customer names;
- service names entered by users;
- product names and product codes;
- descriptions, notes, reasons, and remarks written by users;
- supplier, employee, and package names entered by users.

Do not change domain enums, database values, or public data contracts only to
make labels friendlier. The change should happen through presenter/view labels,
mapping helpers, translation files, or dedicated formatter services.

## Owner Clue

Owner noted that transaction UI wording and any later UI-facing transaction text
that is not user input should use easy Indonesian wherever possible.

Examples of likely problem categories:

- internal words such as `revision`, `snapshot`, `allocation`, `outstanding`,
  `component`, `source`, `gross`, `net`;
- mixed English/Indonesian labels;
- system status copy that reads like code or accounting language;
- report/export headers that expose internal implementation words.

## Problem Analysis

The application has grown through payment/refund/revision/inventory lifecycle
work. Many labels were created to be technically precise while debugging the
domain, but owner-facing UI now needs a clearer reporting/operational language
layer.

This is a broad presentation-language scope. It should not be bundled into
`0046` because `0046` is specifically about report screen/PDF shape and detail
visibility.

## Decision

Create a separate language-normalization backlog before implementation.

The implementation should:

- preserve user-entered values exactly;
- preserve internal enums and database values;
- map internal/domain terms to simple Indonesian only at presentation
  boundaries;
- add tests against rendered UI/export labels for each touched surface;
- proceed slice by slice, starting from transaction/report surfaces with the
  highest owner visibility.

## Initial Wording Direction

Potential owner-facing replacements, to be decided per slice:

- `revision` -> `perubahan nota` or `versi nota`;
- `snapshot` -> `data saat itu`;
- `allocation` -> `pembayaran tercatat`;
- `outstanding` -> `sisa tagihan` or `sisa yang belum dibayar`;
- `gross` -> `total sebelum pengurang`;
- `net` -> `total bersih`;
- `component` -> `bagian pembayaran`;
- `source` -> `asal data`;
- `refund due` -> `pengembalian yang perlu dibayar`;
- `refund paid` -> `pengembalian yang sudah dibayar`;
- `open/close` status -> use existing final UI terms from domain rules where
  applicable.

## Proposed Step Order

1. Inventory transaction/report labels already touched by `0046`.
2. Build a target label map for transaction note/detail/payment/refund surfaces.
3. Patch one page family at a time with rendered assertions.
4. Keep user input values unchanged and explicitly tested.
5. Avoid changing storage/domain enum values unless a separate owner decision
   requires it.

## Proof Needed Before Implementation

- Source scan of current transaction/report labels.
- Rendered UI tests proving system-generated labels are easy Indonesian.
- Regression tests proving user-entered values are not translated or modified.
