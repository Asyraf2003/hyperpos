# Error Log Remediation Definition of Done

- Status: Planning DoD.
- Scope: Definition of Done for execution of `docs/04_lifecycle/error_log/`.
- Non-goal: this document is not a source patch and is not a claim that the error log is finished.

## DoD Principles

- An issue is not complete just because:
  - the document says `Patched`
  - the commit message sounds convincing
  - the source file changed
  - `php -l` passed
  - a UI button was hidden
  - a new test was written but not run
  - one happy path passed
- The DoD must be verifiable from source, tests, command output, and documentation.

## Minimum DoD Per Issue

- Every issue must have:
  - error-log path
  - document status at intake
  - confidence status: `trusted`, `weak`, `contradicted`, or `unknown`
  - final root cause
  - production source map
  - affected layer
  - upstream dependency
  - downstream dependency
  - RED proof or a reason RED is impossible
  - minimal patch boundary
  - focused test target
  - wider regression target
  - UI Blade impact decision
  - native JS impact decision
  - security / authorization impact decision
  - audit / log / redaction impact decision
  - closure proof
  - residual gap
  - handoff note
- An issue may not be closed if any of the items above is missing without a reason.

## Source / Backend DoD

- Source / backend DoD is satisfied only if:
  - the related production files have been inspected
  - the related route / controller / middleware / use case / service / adapter / query has been mapped
  - the root cause matches current source
  - the patch is within the correct boundary
  - the patch is minimal and not a broad refactor without need
  - the source does not conflict with the latest relevant ADR
  - the source does not remove audit/history without a domain decision
  - the source does not trust client input for:
    - price basis
    - row state
    - note state
    - actor id
    - capability performer
    - MIME type
    - filename
    - path
    - URL
    - date-window authorization
    - payment/refund allocation total
  - finance mutation happens inside the correct transaction boundary
  - payment / revision concurrency uses a consistent lock protocol when relevant
  - current vs historical row boundaries are clear
  - legacy vs component payment compatibility is clear
  - refund lifecycle cannot finalize a state without financial proof
  - terminal states such as `closed` / `refunded` are handled according to policy

## UI Blade DoD

- UI Blade DoD is satisfied only if:
  - all affected view paths are recorded
  - action button / link / form matches `can_*` flags or policy data
  - backend still rejects direct requests
  - UI hiding is not claimed as a security boundary
  - user-controlled text is escaped
  - JSON inside `<script>` uses safe encoding such as `@json`, `Js::from`, or correct `JSON_HEX` flags
  - no raw `json_encode` is used unsafely in a script context
  - unsafe strings such as `</script>` do not appear literally in a relevant response
  - URL-like attributes such as `href`, `src`, and `action` use validated safe URLs
  - `javascript:`, `data:`, external URLs, and protocol-relative URLs are rejected or fall back safely
  - global count/stat data does not leak to cashier users
  - Blade does not introduce inline PHP raw blocks for user-controlled data
  - locked UI terms remain intact if the slice touches UI:
    - Note
    - Case
    - Detail
    - Not Paid Yet
    - Paid
    - Canceled
    - Refunded

## Native JS DoD

- Native JS DoD is satisfied only if:
  - affected JS files or inline config are recorded
  - JS is only progressive enhancement
  - form fallback remains safe without JS
  - the server remains the source of truth
  - JS is not the only guard
  - JS does not trust hidden fields for financial or security decisions
  - JSON config is safe from script breakout
  - there is no `eval`, dynamic script injection, or string-to-code behavior
  - there is no `innerHTML` from untrusted input without an ADR-approved sanitizer
  - selected-row behavior is still validated server-side
  - return/back URL is still validated server-side
  - client-side price basis, row ID, date, amount, MIME, or capability is not treated as authority
  - a negative test or response test proves a dangerous payload is not executable when relevant

## Security DoD

- Security DoD is satisfied only if:
  - guest, cashier, admin without capability, and admin with capability are tested according to route
  - direct route requests are tested, not only UI flow
  - cashier date-window uses server-side date/application timezone
  - client-submitted date is not treated as authorization truth
  - admin read access and admin mutation access are separated
  - transaction-sensitive admin mutation requires transaction capability
  - capability toggle requires auth / admin / CSRF
  - audit performer comes from the authenticated session
  - refund / payment / workspace / procurement mutation has access gate and domain eligibility
  - private storage is not exposed through public helper, symlink, `asset()`, or raw public path
  - private proof attachment is served through an authenticated route / policy
  - attachment MIME is detected server-side
  - risky or unknown attachments are forced to download
  - response attachment uses `X-Content-Type-Options: nosniff`
  - filename output is sanitized and uses safe `Content-Disposition`
  - unsafe URLs are rejected or safely fallen back
  - output context matches ADR-0020
  - authorization boundary matches ADR-0019
  - concurrency boundary matches ADR-0022 when payment / revision allocation is involved
  - seeder credential safety is not claimed finished in the main workflow

## Audit / Logging / Redaction DoD

- Audit / log / redaction DoD is satisfied only if:
  - sensitive mutation records actor, role, target resource, action, and timestamp
  - before / after state is recorded when available and relevant
  - manual reason remains required for domain actions that need it
  - capability toggle records actor, target actor, before state, after state, action, and timestamp
  - denied sensitive attempts are logged when the policy or project requires it
  - logs do not store secrets, tokens, raw private paths, or unnecessary proof data
  - audit performer cannot be forged through request body input
  - attachment serving does not expose absolute local paths
  - error responses do not expose private storage internals

## DoD Test

- Test DoD is satisfied only if:
  - RED characterization exists before the patch
  - the RED failure matches the root cause, not a fixture error
  - targeted GREEN passes after the patch
  - focused blast-radius tests pass for sensitive clusters
  - wider regression passes for the affected domain
  - important commands and outputs are recorded
  - assertion counts are recorded when available
  - no-mutation assertions exist for rejected requests
  - no-leak assertions exist for XSS / disclosure / storage issues
  - route-list proof exists for route / middleware issues
  - grep / static negative search exists for Blade / JS / storage patterns when relevant
  - concurrency issues have lock/source proof and, if possible, a true parallel stress test
  - tests are not weakened just to make the patch look green
  - syntax checks are not treated as behavior proof
  - missing vendor / dependency means verification gap, not pass

## DoD Docs

- Docs DoD is satisfied only if:
  - `docs/04_lifecycle/error_log/<issue>.md` is updated after proof, not before
  - the status uses language that does not overstate proof
  - the final root cause is recorded
  - source reality is recorded
  - RED proof is recorded
  - GREEN proof is recorded
  - focused proof is recorded
  - wider proof is recorded when available
  - UI Blade impact is recorded
  - native JS impact is recorded
  - security impact is recorded
  - audit / log / redaction impact is recorded
  - residual gap is recorded
  - conflict with other documents is recorded
  - if the owner accepts deferment, the defer reason and scope are recorded
  - commit hash is recorded only if the commit truly exists
  - docs closure does not claim the seeder is finished

## DoD Final Closure

- Final closure of the entire error log may happen only if:
  - all 29 error logs have entered the sequence
  - all issues have a final trust status
  - all weak / contradicted issues are resolved or deferred with owner acceptance
  - all active slices have targeted / focused proof
  - all sensitive clusters have wider regression proof
  - source/docs conflicts are resolved
  - ADR conflicts are resolved
  - UI Blade and native JS review are complete
  - security authorization review is complete
  - audit / log / redaction review is complete
  - the final global verification pass is complete
  - global blockers such as PHPStan, audit-lines, audit-blade, contract audit, or full test failures are not active
  - the documentation closure packet is complete
  - seeder remains a future scope unless the owner opens a separate seeder scope

## Anti-Claim Rules

- You may not claim:
  - fixed without targeted proof
  - strictly fixed without focused proof for a sensitive issue
  - secure without direct-request authorization proof
  - verified if the test did not run
  - globally verified if the full gate failed
  - safe if only the UI button was hidden
  - XSS fixed if the rendered response was not tested or another sink was not checked
  - private storage safe if deployment / public helper / symlink was not checked
  - concurrency fixed if only a single-request test exists and the lock protocol was not proven
  - refund safe if row eligibility, parent note eligibility, and finalization proof are incomplete
  - payment safe if legacy / component allocation and outstanding were not tested
  - audit safe if `actor` / `performed_by` is still client-controlled
  - seeder fixed inside the main workflow for this session
  - all error logs are closed while weak / contradicted / unverified issues remain

## Stop Condition for Closure

- Stop closure if:
  - proof is missing
  - the source differs from the document
  - the test did not run
  - wider regression failed
